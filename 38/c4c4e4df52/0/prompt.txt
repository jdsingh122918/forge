# Comprehensive PR Review

Run a comprehensive pull request review using multiple specialized agents, each focusing on a different aspect of code quality.

**Review Aspects (optional):** ""

## Review Workflow:

1. **Determine Review Scope**
   - Check git status to identify changed files
   - Parse arguments to see if user requested specific review aspects
   - Default: Run all applicable reviews

2. **Available Review Aspects:**

   - **comments** - Analyze code comment accuracy and maintainability
   - **tests** - Review test coverage quality and completeness
   - **errors** - Check error handling for silent failures
   - **types** - Analyze type design and invariants (if new types added)
   - **code** - General code review for project guidelines
   - **simplify** - Simplify code for clarity and maintainability
   - **all** - Run all applicable reviews (default)

3. **Identify Changed Files**
   - Run `git diff --name-only` to see modified files
   - Check if PR already exists: `gh pr view`
   - Identify file types and what reviews apply

4. **Determine Applicable Reviews**

   Based on changes:
   - **Always applicable**: code-reviewer (general quality)
   - **If test files changed**: pr-test-analyzer
   - **If comments/docs added**: comment-analyzer
   - **If error handling changed**: silent-failure-hunter
   - **If types added/modified**: type-design-analyzer
   - **After passing review**: code-simplifier (polish and refine)

5. **Launch Review Agents**

   **Sequential approach** (one at a time):
   - Easier to understand and act on
   - Each report is complete before next
   - Good for interactive review

   **Parallel approach** (user can request):
   - Launch all agents simultaneously
   - Faster for comprehensive review
   - Results come back together

6. **Aggregate Results**

   After agents complete, summarize:
   - **Critical Issues** (must fix before merge)
   - **Important Issues** (should fix)
   - **Suggestions** (nice to have)
   - **Positive Observations** (what's good)

7. **Provide Action Plan**

   Organize findings:
   ```markdown
   # PR Review Summary

   ## Critical Issues (X found)
   - [agent-name]: Issue description [file:line]

   ## Important Issues (X found)
   - [agent-name]: Issue description [file:line]

   ## Suggestions (X found)
   - [agent-name]: Suggestion [file:line]

   ## Strengths
   - What's well-done in this PR

   ## Recommended Action
   1. Fix critical issues first
   2. Address important issues
   3. Consider suggestions
   4. Re-run review after fixes
   ```

# Type Design Analysis: `feat/libsql-turso-migration` Branch

This analysis covers the new and modified types introduced by the libsql/Turso migration PR, comparing against `main`. The PR splits the monolithic `src/factory/db.rs` into a well-organized module tree (`db/mod.rs`, `db/agents.rs`, `db/issues.rs`, `db/pipeline.rs`, `db/projects.rs`, `db/settings.rs`, `db/migrations.rs`) and introduces a proper migration framework. It also splits `pipeline.rs` into `pipeline/mod.rs`, `pipeline/execution.rs`, `pipeline/git.rs`, and `pipeline/parsing.rs`.

## Type: `DbHandle` (`src/factory/db/mod.rs`)

### Invariants Identified
1. **Post-construction validity**: After construction, the database is guaranteed to have run all migrations and configured appropriate PRAGMAs for the connection mode.
2. **Shared connection invariant**: A single `Connection` (`Arc<Connection>`) is reused for all operations, preventing the libsql in-memory database isolation bug where each `connect()` call creates a separate database.
3. **Mode consistency**: The `DbMode` field determines whether `sync()` is meaningful (only for `EmbeddedReplica`) and whether PRAGMAs are applied (only for `Local`).
4. **Graceful degradation**: `new_remote_replica` falls back to pure `Remote` mode if the embedded replica handshake or initial sync fails, cleaning up stale files.
5. **Immutability of configuration**: Once constructed, `db`, `conn`, and `mode` are never mutated.

### Ratings
- **Encapsulation**: 8/10
  `DbMode` is private (`enum DbMode` without `pub`), which is excellent -- callers cannot inspect or depend on the connection mode. The `Database` handle is wrapped in `Arc` and never exposed. The `conn()` method returns `&Connection`, which is appropriate since libsql connections are internally synchronized. The one weakness is that `conn()` returns a raw `&Connection`, allowing callers to bypass the convenience methods and run arbitrary SQL. This is by design for flexibility, but it does mean external code could violate database invariants (e.g., drop tables, run raw DDL).

- **Invariant Expression**: 7/10
  The three constructors (`new_local`, `new_remote_replica`, `new_remote`) cleanly map to the three `DbMode` variants, making the construction path self-documenting. The `from_db` helper centralizes the pragma-then-migrate initialization sequence, which is good. However, the type system does not prevent calling `sync()` on a non-replica handle -- it silently no-ops. This is pragmatic but not type-level enforced. The `Arc<Database>` field `db` is kept but never used after construction except by `sync()`, which is a minor design smell.

- **Invariant Usefulness**: 9/10
  The shared-connection invariant directly prevents a real, documented bug (in-memory databases creating separate namespaces per `connect()` call). The migration-on-construction guarantee prevents the application from operating against an un-migrated schema. The fallback chain in `new_remote_replica` prevents hard failures in degraded network conditions.

- **Invariant Enforcement**: 8/10
  All construction paths go through `from_db` (or the equivalent inline code in `new_remote_replica` for the replica case), ensuring PRAGMAs and migrations are always run. There is no public `Default` or manual struct construction possible since the fields are private. The one gap: the inline construction in `new_remote_replica`'s success path duplicates the pragma/migration logic rather than calling `from_db`, creating a maintenance risk if the initialization sequence changes.

### Strengths
- Private `DbMode` enum prevents external coupling to the connection strategy.
- Single shared connection with clear documentation of the safety argument (libsql's internal mutex serialization).
- The fallback chain in `new_remote_replica` is well-structured with proper cleanup of stale replica files.
- Comprehensive convenience methods that delegate cleanly to per-domain submodules.
- Good test coverage of the shared connection visibility invariant.

### Concerns
1. **Duplicated initialization logic**: The `new_remote_replica` success path (lines 81-88) manually creates `conn`, calls `init_pragmas`, and `run_migrations` instead of delegating to `from_db`. If `from_db` gains additional initialization steps in the future, this code path will diverge.
2. **Massive facade surface**: `DbHandle` has approximately 30 convenience methods that are pure pass-throughs. Each follows the exact same pattern of `let conn = self.conn(); module::function(&conn, ...).await`. This is boilerplate-heavy and creates a maintenance burden when adding new queries.
3. **`event_type: &str` in public API**: `create_agent_event` and `insert_agent_event` accept `event_type` as `&str` rather than `&AgentEventType`. The `AgentEventType` enum exists and is used in the model, but the database layer accepts raw strings, deferring validation to the parse inside `create_agent_event`. This creates a stringly-typed seam at the API boundary.
4. **`priority: Option<&str>` in `update_issue`**: Same pattern -- `Priority` enum exists but is not used in the function signature.
5. **`phase_number: &str`**: The `phase_number` parameter in `upsert_pipeline_phase` is a raw string, but the domain uses hierarchical phase identifiers like `"1"`, `"2.1"`. No validation or newtype prevents passing arbitrary strings.

### Recommended Improvements
1. **Eliminate the duplicated initialization path**: Refactor `new_remote_replica`'s success branch to call `from_db` and store the `db` value, rather than inlining the same logic. This would look roughly like:
   ```rust
   Ok(db) => {
       match db.sync().await {
           Ok(_) => {
               eprintln!("[db] Embedded replica synced successfully");
               Self::from_db(db, DbMode::EmbeddedReplica).await
           }
           ...
       }
   }
   ```

2. **Accept typed enums at the public API boundary**: Change `create_agent_event` to accept `event_type: &AgentEventType` and `update_issue` to accept `priority: Option<&Priority>`. Call `.as_str()` internally when binding to SQL. This moves validation to compile-time at the call site.

3. **Consider a trait or macro for the facade methods**: The 30+ convenience methods on `DbHandle` are pure delegation. If the pattern continues to grow, consider either (a) having callers use `db.conn()` directly with the module functions, reducing the facade, or (b) a macro to generate the boilerplate. This is a pragmatic judgment call -- the current approach is readable but scales linearly in maintenance cost.

## Type: Migration System (`src/factory/db/migrations.rs`)

### Invariants Identified
1. **Monotonic version ordering**: The `MIGRATIONS` constant is a compile-time array of `(i64, &str)` tuples ordered by version number. Migrations are applied in order, skipping any already recorded in `_migrations`.
2. **Idempotency**: `run_migrations` can be called multiple times safely -- already-applied migrations are skipped via version comparison.
3. **Bootstrap compatibility**: Pre-migration databases (existing tables but no `_migrations` table) are detected and bootstrapped to the correct version without re-running destructive DDL.
4. **Version ceiling**: Bootstrap detection caps at version 4 because versions 5+ were introduced alongside the migration framework itself.

### Ratings
- **Encapsulation**: 7/10
  All helper functions (`ensure_migrations_table`, `get_schema_version`, `set_schema_version`, `run_from_version`, `bootstrap_existing_db`) are private. Only `run_migrations` is `pub`. The `MIGRATIONS` constant is module-private. The only exposure is that `run_migrations` takes a raw `&Connection`, which is appropriate for its role as infrastructure called from `DbHandle` construction.

- **Invariant Expression**: 6/10
  The version ordering invariant is not enforced by the type system. `MIGRATIONS` is a `&[(i64, &str)]` array where versions must be manually kept in ascending order -- a compile-time assertion or at least a runtime debug-assert would strengthen this. The bootstrap version ceiling of 4 is documented in a comment but is a magic number that could become stale. There is no `Migration` struct -- just a tuple of `(i64, &str)`, which is less expressive than a named type.

- **Invariant Usefulness**: 8/10
  The migration system solves a real problem: coordinating schema evolution across local SQLite, Turso embedded replica, and Turso remote HTTP connections. The bootstrap logic is critical for upgrading existing installations without data loss. The idempotency guarantee is essential for server restarts.

- **Invariant Enforcement**: 7/10
  Migrations are applied within `run_from_version` which iterates and records each version. However, migrations are NOT run inside a transaction -- each `execute_batch(sql)` call is followed by a separate `set_schema_version` call. If a migration partially succeeds (e.g., one statement in a multi-statement batch succeeds but a later one fails), the version is not recorded but the schema is partially modified, leaving the database in an inconsistent state. This is a significant gap.

### Strengths
- Clean, simple design that works across all three database modes.
- Smart bootstrap logic that inspects `sqlite_master` rather than using `PRAGMA table_info` (which does not work over Turso HTTP).
- Excellent test coverage: fresh database, idempotent re-runs, partial migration resume, and all four bootstrap version levels.

### Concerns
1. **No transactional migration execution**: If a migration contains multiple statements and one fails, the schema is left in a partial state with no version recorded. The next `run_migrations` call will re-attempt the same migration, potentially failing again on the already-applied statements (or worse, succeeding with `IF NOT EXISTS` but leaving orphaned columns).
2. **No compile-time or runtime check for version ordering**: If someone inserts a migration with version `3` after version `6` in the array, it would be silently skipped on databases that have already reached version 3.
3. **No `Migration` struct**: Using `(i64, &str)` tuples means there is no place to attach metadata like a migration name or description, and no named fields to prevent accidentally swapping version and SQL.

### Recommended Improvements
1. **Add a debug assertion for version ordering**: At the start of `run_migrations` or as a `const` assertion, verify that `MIGRATIONS` versions are strictly ascending. A simple runtime check in debug builds would catch this:
   ```rust
   debug_assert!(MIGRATIONS.windows(2).all(|w| w[0].0 < w[1].0), "Migrations must be in ascending version order");
   ```

2. **Consider a named struct for migrations**:
   ```rust
   struct Migration {
       version: i64,
       name: &'static str,
       sql: &'static str,
   }
   ```
   This is a minor improvement but adds clarity and prevents tuple-element confusion.

3. **Document the lack of transactional guarantees**: If wrapping each migration in a transaction is not feasible (libsql/Turso may not support DDL in transactions over HTTP), at minimum add a comment explaining this limitation so maintainers write each migration SQL to be idempotent (using `IF NOT EXISTS`, `IF EXISTS`, etc.).

## Type: Domain Model Structs (`Project`, `Issue`, `PipelineRun`, `AgentTeam`, `AgentTask`, `AgentEvent`)

### Invariants Identified
1. **All fields public**: Every struct has all fields `pub`, making them effectively DTOs (data transfer objects).
2. **Typed enums for categorical fields**: `IssueColumn`, `Priority`, `PipelineStatus`, `PhaseStatus`, `AgentRole`, `AgentTaskStatus`, `IsolationStrategy`, `ExecutionStrategy`, `AgentEventType` -- all use typed enums rather than raw strings.
3. **Timestamps as strings**: All timestamp fields (`created_at`, `updated_at`, `started_at`, `completed_at`) are `String` rather than a typed datetime.

### Ratings
- **Encapsulation**: 3/10
  All fields are public. There is no private state, no constructor validation, and no controlled mutation. Any code with access to the struct can set `status` to an arbitrary value, set `created_at` to an empty string, or set `position` to a negative number. These are essentially C-style structs with no encapsulation.

- **Invariant Expression**: 6/10
  The use of typed enums for categorical fields is good and prevents invalid status/column/priority values at the type level.   There is no enforcement beyond what the enum types provide. No `new()` constructors with validation. No builder pattern. No private fields. The structs are constructed directly in the database layer, which is the sole point of implicit validation (SQL constraints serve as the real invariant enforcement).

### Strengths
- Consistent enum design with `as_str()`, `FromStr`, `Display`, and `Serialize/Deserialize` on every categorical enum.
- Excellent documentation with doc comments explaining the purpose and lifecycle of each type.

- Roundtrip tests for every enum.

### Concerns
1. **Stringly-typed timestamps**: `created_at: String` provides no guarantee that the value is a valid ISO 8601 timestamp. A `chrono::DateTime<Utc>` or `chrono::NaiveDateTime` would provide compile-time guarantees and prevent format inconsistencies.
2. **No newtypes for IDs**: `project_id: i64`, `issue_id: i64`, `run_id: i64`, `team_id: i64`, `task_id: i64` are all bare `i64`. It is trivially easy to pass a `project_id` where a `run_id` is expected with no compile-time error.

The database stores them as separate columns, but the Rust type could derive `has_team` rather than storing it.

### Recommended Improvements
1. **Consider newtypes for high-traffic ID types** (low priority, high disruption): Types like `ProjectId(i64)`, `IssueId(i64)`, `RunId(i64)` would prevent ID confusion. However, this is a significant refactor with moderate benefit -- only worth doing if ID confusion bugs have occurred in practice.

2. **Derive `has_team` instead of storing it**: Replace the `has_team` field with a method:
   ```rust
   impl PipelineRun {
       pub fn has_team(&self) -> bool {
           self.team_id.is_some()
       }
   }
   ```
   Use `#[serde(skip)]` or a custom serializer if the JSON API needs the field. This eliminates the source-of-truth duplication.

3. **Consider typed timestamps**: If `chrono` is already a dependency (it is -- used in `agents.rs` for `chrono::Utc::now().to_rfc3339()`), using `chrono::DateTime<Utc>` for timestamp fields would provide type safety. This requires custom serde handling for the SQLite datetime format, so it is a moderate-complexity change.

# Error Handling Audit: `feat/libsql-turso-migration` PR

## Summary

This audit covers the migration from rusqlite to libsql with async support across the factory subsystem. I examined every file changed in this PR for silent failures, inadequate error handling, unjustified fallbacks, and missing error context. Below are the findings, ordered by severity.

## CRITICAL Issues

### Issue 1: Silent fallback from embedded replica to remote HTTP mode hides connection problems

**Location:** `/Users/jdsingh/Projects/AI/forge/src/factory/db/mod.rs`, lines 59-104

**Severity:** CRITICAL

**Issue Description:** The `new_remote_replica` method silently falls back to pure remote HTTP mode when the embedded replica fails to build OR when the initial sync fails. This means a misconfigured `TURSO_DATABASE_URL`, a network blip, or a corrupted local file all get silently swallowed, and the user ends up on a degraded connection mode without being asked. The fallbacks cascade: first try embedded replica, then on ANY failure, try remote HTTP. If the remote HTTP also fails, only THEN does the user see an error.

This is particularly dangerous because:
- The fallback from embedded replica (fast, local reads) to pure remote HTTP (every query goes over the network) represents a **massive performance degradation** that the user has no way to detect aside from an `eprintln` message
- Lines 93-95 use `let _ =` to discard errors from file cleanup, which is acceptable for cleanup but compounds the silent nature of the overall fallback
- There is no structured logging, no metric emission, and no way for monitoring to detect this happened

**Hidden Errors:** Corrupted local replica files, DNS resolution failures, expired auth tokens, TLS certificate issues, disk permission errors -- all silently swallowed and replaced with a degraded mode.

**User Impact:** Users think they are running with a fast local replica but are actually hitting Turso over HTTP for every query. Performance will be noticeably worse but the cause will be invisible.

**Recommendation:** At minimum, the fallback should be logged at a WARNING level with structured context (e.g., URL, error type). Ideally, the fallback should be configurable: some deployments may prefer to fail hard rather than silently degrade to remote HTTP mode.

# Subagent-Driven Development

Execute plan by dispatching fresh subagent per task, with two-stage review after each: spec compliance review first, then code quality review.

**Core principle:** Fresh subagent per task + two-stage review (spec then quality) = high quality, fast iteration

## Advantages

**vs. Manual execution:**
- Subagents follow TDD naturally
- Fresh context per task (no confusion)
- Parallel-safe (subagents don't interfere)
- Subagent can ask questions (before AND during work)

**vs. Executing Plans:**
- Same session (no handoff)
- Continuous progress (no waiting)
- Review checkpoints automatic

**Efficiency gains:**
- No file reading overhead (controller provides full text)
- Controller curates exactly what context is needed
- Subagent gets complete information upfront
- Questions surfaced before work begins (not after)

**Quality gates:**
- Self-review catches issues before handoff
- Two-stage review: spec compliance, then code quality
- Review loops ensure fixes actually work
- Spec compliance prevents over/under-building
- Code quality ensures implementation is well-built

**Cost:**
- More subagent invocations (implementer + 2 reviewers per task)
- Controller does more prep work (extracting all tasks upfront)
- Review loops add iterations
- But catches issues early (cheaper than debugging later)

## Red Flags

**Never:**
- Start implementation on main/master branch without explicit user consent
- Skip reviews (spec compliance OR code quality)
- Proceed with unfixed issues
- Dispatch multiple implementation subagents in parallel (conflicts)
- Make subagent read plan file (provide full text instead)
- Skip scene-setting context (subagent needs to understand where task fits)
- Ignore subagent questions (answer before letting them proceed)
- Accept "close enough" on spec compliance (spec reviewer found issues = not done)
- Skip review loops (reviewer found issues = implementer fixes = review again)
- Let implementer self-review replace actual review (both are needed)
- **Start code quality review before spec compliance is ✅** (wrong order)
- Move to next task while either review has open issues

**If subagent asks questions:**
- Answer clearly and completely
- Provide additional context if needed
- Don't rush them into implementation

**If reviewer finds issues:**
- Implementer (same subagent) fixes them
- Reviewer reviews again
- Repeat until approved
- Don't skip the re-review

**If subagent fails task:**
- Dispatch fix subagent with specific instructions
- Don't try to fix manually (context pollution)
