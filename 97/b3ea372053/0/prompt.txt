## Section 18: Terminal UI

### Location

`src/ui/mod.rs` — re-exports from three submodules: `dag_progress`, `icons`, and `progress`.

### Public Surface

The module exposes two primary types:

- `DagUI` and `UiMode` from `dag_progress` — for rendering DAG execution state in the terminal
- `OrchestratorUI` from `progress` — for rendering sequential orchestrator progress

The `icons` submodule provides terminal glyphs and symbols used by the other two renderers.

### Design Intent

The UI layer is split by execution model. Sequential runs (the standard `forge run` path) use `OrchestratorUI`, which tracks the current phase and iteration and renders a line-oriented progress display suitable for streaming to a TTY. DAG/swarm runs use `DagUI`, which must represent multiple concurrently executing nodes — requiring a more structured layout that can show per-node state simultaneously.

`UiMode` controls how `DagUI` renders: the same type can emit a full interactive terminal view or a simpler log-friendly mode when not connected to a TTY.

### OrchestratorUI

`OrchestratorUI` is the progress renderer for sequential phase execution. It receives events as the orchestrator runner advances through phases and iterations. Its responsibilities include:

- Displaying the current phase name and iteration count against the iteration budget
- Rendering signals parsed from Claude output (`<progress>`, `<blocker>`, `<pivot>`) in a visually distinct way
- Showing timing information per iteration and per phase
- Indicating phase outcomes (completed via `<promise>DONE</promise>`, budget exhausted, or failed)

### DagUI

`DagUI` is the renderer for swarm/DAG execution, where multiple phases run in parallel. Key behaviors:

- Maintains per-node status keyed by phase name
- Renders a grid or list of nodes, each showing: phase name, current iteration, budget, and status (pending, running, completed, failed, blocked)
- Uses `UiMode` to decide between a full terminal-rewrite loop (interactive TTY) and a line-append mode (CI logs or non-TTY environments)

### Icons Submodule

`icons` centralizes Unicode/ASCII glyphs used across both renderers — checkmarks, spinners, warning symbols, and phase-state indicators — ensuring visual consistency and easy fallback to ASCII-only for environments that cannot render Unicode.

### Relationship to Other Subsystems

The UI layer is purely presentational. The orchestrator runner (`src/orchestrator/runner.rs`) and DAG executor (`src/dag/executor.rs`) are the event sources; the UI modules render their streams. The Factory UI (React, `ui/src/`) is separate and communicates over WebSocket — the terminal UI modules are only used by CLI invocations.

## Section 19: Metrics

### Location

`src/metrics/mod.rs` — defines `MetricsCollector` and re-exports the `queries` submodule.

### Storage Model

Metrics are persisted to SQLite via `DbHandle` (the same libsql-backed handle used by the Factory database). Every `record_*` call issues a SQL write immediately and asynchronously — there is no in-memory accumulator. This makes metrics durable across restarts and directly queryable by the Factory API.

Five tables are written to:

| Table | Granularity | Key columns |
|---|---|---|
| `metrics_runs` | One row per pipeline run | `run_id`, `issue_id`, `success`, `duration_secs`, `phases_total`, `phases_passed`, `started_at`, `completed_at` |
| `metrics_phases` | One row per phase per run | `run_id`, `phase_number`, `phase_name`, `budget`, `outcome`, `iterations_used`, `duration_secs`, diff stats |
| `metrics_iterations` | One row per iteration per phase | token counts, signal counts (`blocker_count`, `pivot_count`, `progress_percent`), `promise_found` |
| `metrics_reviews` | One row per specialist review | `specialist_type`, `verdict`, `findings_count`, `critical_count` |
| `metrics_compactions` | One row per context compaction | `iterations_compacted`, char counts, `compression_ratio` |

### Write API

`MetricsCollector` exposes a lifecycle-paired write API:

- `record_run_started` / `record_run_completed` — INSERT then UPDATE; completion returns an error if no matching run exists, enforcing call ordering.
- `record_phase_started` / `record_phase_completed` — same pattern; completion carries outcome string, iteration count, duration, and git diff statistics (files added/modified/deleted, lines added/removed).
- `record_iteration` — single INSERT per iteration; captures character counts, token counts (both `Option<i32>`), and parsed signal summary.
- `record_review` — INSERT per specialist verdict.
- `record_compaction` — INSERT per context compaction event with compression ratio.

### Read API (Queries)

Five read methods aggregate over a caller-specified trailing window in days:

- **`summary_stats(days)`** — `SummaryStats`: total/successful runs, success rate, avg duration, total phases, avg iterations per phase.
- **`phase_stats_by_name(days)`** — `Vec<PhaseNameStats>`: per phase name, run count, avg iterations, avg duration, budget utilization (`iterations_used / budget`, guarded against zero-budget phases), success rate. Ordered by run count descending.
- **`review_stats(days)`** — `Vec<ReviewStats>`: per specialist type, total reviews, pass rate, avg findings, avg critical findings.
- **`recent_runs(limit)`** — `Vec<RunSummary>` ordered newest first; used by Factory UI dashboard.
- **`token_usage(days)`** — `Vec<TokenDailyUsage>` grouped by calendar day with summed input/output tokens; enables a token consumption chart.

### Error Handling

The UPDATE-based completion methods check `rows_affected == 0` and bail with an informative message, catching lifecycle ordering bugs at the point of the bad call. All methods use `.context()` on every database call.

### Testing

11 unit tests, all using an in-memory `DbHandle`. Coverage includes full run/phase/iteration lifecycles, review and compaction inserts, all five aggregate queries (including empty-database and zero-budget-phase edge cases), and the error path for completing a nonexistent run.

## Section 20: Factory Data Model

### Overview

The Factory's data model lives entirely in `src/factory/models.rs` and is backed by a SQLite/Turso database accessed through `src/factory/db/mod.rs`. The model file defines all domain types — plain Rust structs and enums — while the database layer provides a typed façade that routes each operation to a dedicated sub-module.

### ID Types

Every entity uses a newtype wrapper over `i64` generated by the `define_id_type!` macro:

```rust
define_id_type!(ProjectId);
define_id_type!(IssueId);
define_id_type!(RunId);
define_id_type!(TeamId);
define_id_type!(TaskId);
define_id_type!(PhaseId);
define_id_type!(EventId);
```

Each ID type derives `Copy`, `Hash`, implements `Display`, and is transparent in JSON (serialises as the bare integer). This prevents mixing up foreign keys at compile time without any runtime overhead.

### Core Entities

**Project** — a registered local codebase. Stores `name`, `path` (absolute filesystem path), and optional `github_repo` (`owner/repo` slug for auto-PR creation).

**Issue** — a Kanban work card. Key fields:
- `column: IssueColumn` — the current Kanban column (`Backlog | Ready | InProgress | InReview | Done`).
- `description: String` — the full spec text; this becomes the agent's prompt.
- `priority: Priority` — `Low | Medium | High | Critical`.
- `labels: Vec<String>` — free-form tags.
- `github_issue_number: Option<i64>` — sync link to a GitHub issue.

Moving an issue to `IssueColumn::InProgress` is the external trigger that launches a `PipelineRun`.

**PipelineRun** — one execution of `forge` against an issue. Tracks:
- `status: PipelineStatus` — state machine: `Queued → Running → Completed | Failed | Cancelled`.
- `phase_count`, `current_phase`, `iteration` — live progress counters updated while running.
- `branch_name` — the git branch created for this run (e.g. `forge/issue-42`).
- `pr_url` — populated after a successful `gh pr create`.
- `team_id` / `has_team` — links to an `AgentTeam` when multi-agent mode is active.

`PipelineStatus::is_terminal()` returns `true` for `Completed | Failed | Cancelled`.

**PipelinePhase** — one named phase within a run. Maps to phases in `.forge/spec.md`. Tracks `phase_number` (hierarchical, e.g. `"2.1"`), `phase_name`, `status: PhaseStatus`, `iteration`, and `budget`. Upserted on every iteration tick.

### Agent Team Entities

When the planner decomposes an issue into parallel work, the following hierarchy is created:

```
AgentTeam (1)
  └── AgentTask (N)
        └── AgentEvent (M)
```

**AgentTeam** — captures the planner's output: `strategy: ExecutionStrategy` (`Parallel | Sequential | WavePipeline | Adaptive`) and `isolation: IsolationStrategy` (`Worktree | Container | Hybrid | Shared`), plus a `plan_summary` narrative.

**AgentTask** — an atomic unit of work assigned to a `role: AgentRole` (`Planner | Coder | Tester | Reviewer | BrowserVerifier | TestVerifier`). The `wave: i32` field controls dependency ordering — all tasks in wave N complete before wave N+1 starts. `depends_on: Vec<TaskId>` carries explicit inter-task dependencies within a wave. Isolation state is recorded in `worktree_path` and `container_id`.

**AgentEvent** — a real-time log entry emitted by an agent: `event_type: AgentEventType` (`Thinking | Action | Output | Signal | Error`), `content`, and optional `metadata: serde_json::Value` for structured `Signal` events. Events are written to the DB and also broadcast over WebSocket.

### Auxiliary Enums

- `SignalType` — `Progress | Blocker | Pivot` — mirrors the `<progress>`, `<blocker>`, `<pivot>` XML tags in agent output.
- `VerificationType` — `Browser | TestBuild` — reserved for future verification task dispatch.
- `FileAction` — `Created | Modified | Deleted` — used in `WsMessage::PipelineFileChanged`.

All enums implement `as_str()`, `Display`, and `FromStr` so they round-trip cleanly through SQLite TEXT columns and JSON. `#[serde(rename_all = "snake_case")]` ensures JSON uses lowercase snake_case variants.

### View Types

Three composite structs are returned by read endpoints:

- `BoardView { project, columns: Vec<ColumnView> }` — the full Kanban board for `GET /api/projects/:id/board`. Each `ColumnView` holds `Vec<IssueWithStatus>` pairing an issue with its active (non-terminal) run.
- `IssueDetail { issue, runs: Vec<PipelineRunDetail> }` — returned by `GET /api/issues/:id`. Each `PipelineRunDetail` flattens a `PipelineRun` with its `phases: Vec<PipelinePhase>`.
- `AgentTeamDetail { team, tasks }` — returned by `GET /api/runs/:id/team`.

### Database Layer (`src/factory/db/mod.rs`)

`DbHandle` wraps a `libsql` connection in three modes:
- `Local` — plain SQLite file (dev/offline).
- `EmbeddedReplica` — Turso replica with local file cache; syncs every 60 seconds. Falls back to `Remote` if the initial sync fails.
- `Remote` — pure HTTP to Turso; no local cache.

A single `Arc<Connection>` is shared across all calls (libsql serialises access internally). On construction, `from_db()` runs `init_pragmas` (WAL mode, 5 s busy timeout, foreign keys — local only) and then `migrations::run_migrations`.

The public API is a flat set of `async fn` methods on `DbHandle` — one per operation — that delegate to five sub-modules:

| Sub-module | Responsibility |
|---|---|
| `projects` | CRUD for `Project` |
| `issues` | CRUD + move + board/detail queries for `Issue` |
| `pipeline` | CRUD + progress updates for `PipelineRun` and `PipelinePhase` |
| `agents` | CRUD for `AgentTeam`, `AgentTask`, `AgentEvent` |
| `settings` | Key-value store for application settings |

Notable operations:
- `move_issue(id, column, position)` — updates column and position in one call; the API handler calls this and then broadcasts an `IssueMoved` WebSocket message.
- `recover_orphaned_runs()` — called on startup to transition any `Running` runs left by a crash back to `Failed`.
- `upsert_pipeline_phase()` — insert-or-update phase progress rows keyed on `(run_id, phase_number)`.
- `cancel_pipeline_run()` — sets status to `Cancelled` and records `completed_at`.

## Section 21: Factory Pipeline

### Overview

`src/factory/pipeline/` is the execution engine that turns a Kanban issue into running code. The directory contains four files:

| File | Responsibility |
|---|---|
| `mod.rs` | `PipelineRunner`, agent-team orchestration, auto-promote policy |
| `execution.rs` | Low-level process/container spawning, stdout streaming |
| `git.rs` | Branch creation, worktree management, PR creation |
| `parsing.rs` | `StreamJsonEvent` parsing for `claude --print --output-format stream-json` |

### Entry Point: `PipelineRunner::start_run`

`start_run` is called by the API handler when an issue is moved to `InProgress`. It immediately updates the run to `Running` and moves the issue column, then spawns a `tokio::spawn` background task so the HTTP response returns instantly.

The background task follows this sequence:

```
1. Acquire git lock → create_git_branch (forge/issue-<id>-<slug>)
2. Construct Planner + AgentExecutor
3. execute_agent_team(...)
   ├── Ok(TeamExecuted(summary))  → use summary directly
   ├── Ok(FallbackToForge)        → run_forge_fallback(...)
   └── Err(_)                     → run_forge_fallback(...)  (unexpected failure)
4. On success: acquire git lock → create_pull_request → update DB → broadcast PipelineCompleted
5. On failure: update DB to Failed → broadcast PipelineFailed
```

The git lock (`GitLockMap` — a `DashMap<String, Arc<Mutex<()>>>` keyed on project path) serialises git-mutating operations per project while allowing concurrent runs across different projects.

### Agent Team Path (`execute_agent_team`)

This is the primary execution path for non-trivial issues.

**Step 1 — Planning.** `Planner::plan(title, description, labels)` calls an LLM to decompose the issue into a `PlanResponse`. If planning fails, `PlanResponse::fallback()` produces a single-task sequential plan. If the plan has exactly one task with `Sequential` strategy, the function returns `PlanOutcome::FallbackToForge` immediately, signalling the caller to use the simpler forge pipeline instead.

**Step 2 — DB persistence.** `AgentTeam` and all `AgentTask` records are written to the database. `depends_on` indices from the plan are resolved to real `TaskId` values by indexing into the already-created `db_tasks` vector.

**Step 3 — Broadcast.** A `WsMessage::TeamCreated` message is sent to all connected WebSocket clients so the UI can render the task board immediately.

**Step 4 — Wave execution loop.** Tasks are grouped by `wave` number (0, 1, 2, ...). For each wave:

1. **Worktree setup** — tasks with `IsolationStrategy::Worktree` get a dedicated `git worktree` via `task_runner.setup_worktree()`. If setup fails in a single-task wave, execution falls back to the shared directory. If it fails in a multi-task wave, the entire run aborts (shared directory is unsafe for parallel agents).

2. **Parallel execution** — each task is launched as a separate `tokio::spawn`. `task_runner.run_task()` (implemented by `AgentExecutor`) drives `claude --print` or `forge` inside the task's working directory, streaming events back over WebSocket.

3. **Merge under lock** — after all tasks in the wave complete, the per-project git lock is acquired and each task's branch is merged back into the base branch (`base_branch` defaults to `"main"`). `WsMessage::MergeStarted` and `WsMessage::MergeCompleted` are broadcast; if any merge produces a conflict the wave aborts.

4. **Worktree cleanup** — all worktrees are removed regardless of merge outcome.

5. **Failure gate** — if any task failed, the wave aborts with an error and the run transitions to `Failed`.

### Forge Fallback Path (`run_forge_fallback`)

Used for simple issues (single-task plan) or when the agent team path fails. Steps:

1. If the project has a `.forge/` directory but no `phases.json`, call `auto_generate_phases`: write a spec to `.forge/spec.md` from the issue title/description and run `forge generate` to produce `phases.json`.

2. Build the execution command:
   - `.forge/phases.json` present → `forge swarm --max-parallel 4 --fail-fast`
   - No phases → `claude --print --verbose --output-format stream-json --dangerously-skip-permissions "<prompt>"`

3. Execute via `execute_pipeline_streaming` (local subprocess) or `execute_pipeline_docker` (Docker sandbox), depending on whether a `DockerSandbox` is configured.

### Streaming Execution (`execution.rs`)

`execute_pipeline_streaming` spawns the command with `stdout: Stdio::piped()`, stores the child handle in `running_processes` for cancellation, then reads stdout line by line:

- Lines matching the DAG executor `PhaseEvent` JSON format → `process_phase_event` (upserts `PipelinePhase` rows, broadcasts `PipelinePhaseStarted` / `PipelinePhaseCompleted`).
- Lines matching a simple progress JSON struct → `update_pipeline_progress` + broadcast `PipelineProgress`.
- All other non-empty lines → `parse_stream_json_line` to classify as `Text`, `Thinking`, `ToolStart`, `ToolEnd`, or `Skip`:
  - `Text` / `Thinking` — throttled to one broadcast per 200 ms.
  - `ToolStart` — broadcast `PipelineOutputEvent` with `content_type: "tool_start"` plus legacy `PipelineOutput`; also checks `extract_file_change` to emit `PipelineFileChanged` for file-editing tool calls.
  - `ToolEnd` — broadcast with `content_type: "tool_end"`.

`execute_pipeline_docker` mirrors the same logic but uses `DockerSandbox::run_pipeline` to start a container, reads log lines from a channel, and enforces a configurable timeout. It checks for OOM kill on completion.

### Cancellation

`PipelineRunner::cancel(run_id)` looks up the `RunHandle` for the given run:
- `RunHandle::Process(child)` → `child.kill()`.
- `RunHandle::Container(id)` → `sandbox.stop(id)`.

After killing, it calls `cancel_pipeline_run` on the DB (sets status to `Cancelled`) and moves the issue back to `Ready`.

### Auto-Promote Policy

`AutoPromotePolicy::decide(pipeline_succeeded, has_review_warnings, arbiter_proceeded, fix_attempts)` encodes post-run column routing:
- Failed run → `PromoteDecision::Failed` (issue stays in `InProgress`).
- Succeeded with warnings or arbiter overrides or multiple fix cycles → `InReview`.
- Clean success → `Done`.

In the current `start_run` implementation, success always moves the issue to `InReview` (the arbiter/review path is not yet wired into the main execution loop), leaving `AutoPromotePolicy` available for future use.

### Status Transitions

Two free functions enforce invariants:
- `is_cancellable(status)` — returns `true` only for `Queued | Running`.
- `is_valid_transition(from, to)` — allows only the five legal state-machine edges: `Queued → Running`, `Queued → Cancelled`, `Running → Completed | Failed | Cancelled`.

## Section 6: The Phase Model

**File:** `src/phase.rs`

### Purpose

A Phase is Forge's unit of work. Every `forge run` execution is a sequence of phases; every parallel swarm node executes a phase. `phase.rs` defines what a phase *is*, what constraints it carries, and how phases are loaded from JSON config. It is a pure data-definition module — no async, no I/O beyond loading from disk.

### Key Types

**`Phase`** — the central struct. Every field is serializable so phases can round-trip through JSON checkpoints and the factory database.

| Field | Type | Meaning |
|---|---|---|
| `number` | `String` | Identifier like `"01"`, `"02"`. Used in log filenames and dependency edges. |
| `name` | `String` | Human-readable label shown in the UI. |
| `promise` | `String` | The exact text Claude must emit inside `<promise>…</promise>` to signal completion. |
| `budget` | `u32` | Maximum iterations allowed. If Claude has not found the promise after this many tries, the phase is marked `MaxIterationsReached`. |
| `depends_on` | `Vec<String>` | Phase numbers that must complete first. Used by the DAG scheduler (`src/dag/`). |
| `skills` | `Vec<String>` | Names of reusable skill files appended to the prompt. |
| `permission_mode` | `PermissionMode` | Controls whether iterations are auto-approved and whether writes are blocked. |
| `parent_phase` | `Option<String>` | Set only on sub-phases; `None` for top-level phases. |
| `reviews` | `Option<PhaseReviewSettings>` | Optional post-phase quality gate configuration. |
| `council` | `Option<bool>` | Per-phase override for council (multi-worker) execution. |

**`SubPhase`** — a dynamically spawned child phase. When Claude discovers its scope is larger than expected, it emits a `<spawn-subphase>` signal (see Section 7). The orchestrator converts that signal into a `SubPhase` and executes it immediately within the parent's remaining budget. Sub-phase numbers follow a dotted scheme: parent `"05"` spawns `"05.1"`, `"05.2"`, etc. `SubPhase::to_phase()` converts a `SubPhase` into a full `Phase` for execution, inheriting the parent's settings.

**`SubPhaseStatus`** — tracks lifecycle: `Pending → InProgress → Completed | Failed | Skipped`.

**`PhaseReviewSettings` / `PhaseSpecialistConfig`** — configure which review specialists run after phase completion, whether they run in parallel, and whether any of them *gate* (block) phase completion if they reject.

**`PhasesFile`** — the top-level struct for `phases.json`. Contains a `Vec<Phase>` plus optional global settings. `load_phases()` deserializes this from disk.

### The Promise Contract

The promise is the only way for a phase to declare itself done. Claude must emit `<promise>DONE</promise>` (where `DONE` matches `phase.promise`). The runner scans each iteration's output for this exact tag. If not found before the budget is exhausted, the phase fails. This is intentionally binary — progress signals (Section 7) exist for visibility only and do not affect the pass/fail outcome.

### Connections

- `src/orchestrator/runner.rs` — reads `phase.budget`, `phase.promise`, `phase.permission_mode`, `phase.skills`.
- `src/dag/scheduler.rs` — reads `phase.depends_on` and `phase.number` to build the DAG.
- `src/gates/mod.rs` — receives a `&Phase` to decide approval strategy per `permission_mode`.
- `src/review/specialists.rs` — receives `phase.reviews` to determine which specialists to invoke.

## Section 7: Signals & Stream Parsing

**Files:** `src/signals/mod.rs`, `src/signals/types.rs`, `src/stream/mod.rs`

### Purpose

Two distinct output channels from Claude are handled here:

1. **Signals** (`src/signals/`) — structured XML-like tags Claude embeds in its text to communicate intent: progress percentage, blockers, strategy pivots, sub-phase spawn requests.
2. **Stream** (`src/stream/`) — the JSON event stream the Claude CLI emits on stdout when run with `--output-format stream-json`. This is the raw transport layer that delivers text, tool calls, and session metadata.

### Signal Types (`src/signals/types.rs`)

All four signal types share a `timestamp: DateTime<Utc>` field and are collected into `IterationSignals` after each invocation.

**`ProgressSignal`** — emitted as `<progress>50%</progress>`. Carries `percentage: u8` (0–100). Used by the gate system and UI to show partial completion. `is_complete()` returns true at 100%. Does not affect the promise check.

**`BlockerSignal`** — emitted as `<blocker>description</blocker>`. Signals Claude is stuck. Has `acknowledged: bool`; unacknowledged blockers are surfaced in the `IterationFeedback` injected into the next iteration's system prompt.

**`PivotSignal`** — emitted as `<pivot>new approach</pivot>`. Signals a strategy change. The runner renders this as a `## STRATEGY CHANGE` directive in the next iteration's feedback, explicitly instructing Claude to follow the new approach and not repeat the old one.

**`SubPhaseSpawnSignal`** — emitted as `<spawn-subphase>{ JSON }</spawn-subphase>`. The body is a JSON object with `name`, `promise`, `budget`, and optional `reasoning`/`skills`. The gate system checks whether the requested budget fits within the parent's remaining budget before approval. This is the mechanism by which Claude can dynamically extend its own scope.

**`IterationSignals`** — the aggregate container. Key methods: `has_signals()`, `latest_progress()`, `has_unacknowledged_blockers()` / `acknowledge_all_blockers()`, `merge()` (folds sub-phase results back), `summary()` (one-line description for logging).

### Stream Events (`src/stream/mod.rs`)

The `StreamEvent` enum mirrors the Claude CLI's line-per-JSON format:

- `StreamEvent::Assistant { message }` — Claude's turn; `AssistantMessage.content` is `Vec<ContentBlock>`
- `ContentBlock::Text { text }` — prose output (where signal tags appear)
- `ContentBlock::ToolUse { name, input, id }` — a tool invocation
- `StreamEvent::Result { subtype, result, is_error }` — terminal event carrying token usage
- `StreamEvent::System { subtype }` — session initialization, carries `session_id`

The runner reads stdout line by line, parses each line as a `StreamEvent`, accumulates text, captures the `session_id` and token usage, and updates the UI via `describe_tool_use()` and `tool_emoji()`.

### Connections

- `src/orchestrator/runner.rs` — calls `extract_signals(&output)` after each iteration; builds `IterationFeedback` from the resulting `IterationSignals`.
- `src/gates/mod.rs` — `ProgressTracker::update()` takes `Option<u8>` from `signals.latest_progress()`.
- `src/audit/mod.rs` — `IterationAudit.signals` stores the full `IterationSignals`.

## Section 8: Sequential Execution

**Files:** `src/orchestrator/mod.rs`, `src/orchestrator/runner.rs`

### Purpose

The orchestrator owns the `forge run` execution path: one phase at a time, one iteration at a time, with checkpoint recovery, feedback injection, and optional council mode. It is deliberately separate from the DAG scheduler (`src/dag/`) — parallel execution is a different concern.

### Persistence Ownership (from mod.rs)

| Layer | What it persists |
|---|---|
| `orchestrator/state.rs` | Phase completion state — checkpoint recovery |
| `audit/logger.rs` | Audit trail — signals, tool calls, raw output |
| `factory/db.rs` | Factory UI state — pipeline runs, WebSocket events |
| `compaction/tracker.rs` | Session IDs, compaction summaries |

`StateManager` is the source of truth for "which phases are already done." On startup, `forge run` reads the state log and skips completed phases automatically. The audit logger and factory DB are observe-only; they never drive control flow.

### Key Types

**`ClaudeRunner`** — wraps a `Config` and exposes `run_iteration()` and `run_effective_iteration()`. The "effective" variant checks whether council mode is active for the phase and dispatches to either `run_council_iteration()` or `run_iteration_with_context()`.

**`IterationResult`** — value returned by a single Claude invocation: `session: ClaudeSession`, `promise_found: bool`, `output: String`, `signals: IterationSignals`.

**`PromptContext`** — optional extra context injected at prompt generation time. Currently used for compaction summaries.

**`IterationFeedback`** — a builder that assembles the `## ITERATION FEEDBACK` block appended via `--append-system-prompt` on every iteration after the first. Adds (in priority order): iteration status, git change summary, signals, pivot directive. Capped at 4,000 characters to prevent doubling system prompt size. Higher-priority parts appear first so they survive truncation.

### The Iteration Loop

Core flow in `run_iteration_with_context()`:

1. Generate prompt — full spec + skills on first iteration; short continuation prompt when resuming.
2. Write prompt to disk as `phase-{N}-iter-{M}-prompt.md`.
3. Spawn Claude CLI via `tokio::process::Command` with `--output-format stream-json`, optional `--resume <session_id>`, optional `--append-system-prompt <feedback>`, and `--allowed-tools` from `permission_mode`.
4. Stream stdout line by line, parsing `StreamEvent`, accumulating text, capturing session ID and token usage, updating UI in real time.
5. Check for promise tag in accumulated output.
6. Call `extract_signals(&output)` to find all signal tags.
7. Return `IterationResult`; caller decides whether to continue or terminate.

### Session Resumption

Forge captures `session_id` from `StreamEvent::System` and threads it through iterations. On non-first iterations, a short continuation prompt is sent to avoid re-sending the full context.

### Connections

- `src/phase.rs` — `Phase` drives prompt generation, tool filtering, promise checking.
- `src/gates/mod.rs` — caller passes `IterationResult` back to gate to decide continue/stop.
- `src/tracker/git.rs` — git snapshot before/after each iteration.
- `src/audit/mod.rs` — `IterationAudit` is constructed from `IterationResult` and appended.
- `src/council/` — `run_council_iteration()` delegates to `CouncilEngine::run_phase()`.

## Section 9: Permission Gates

**File:** `src/gates/mod.rs`

### Purpose

Decides whether a phase or iteration is allowed to proceed, and whether to block filesystem writes. Implements the per-phase `PermissionMode` policy. Also detects stale progress in autonomous mode and injects a pivot prompt to force a strategy change.

### Permission Modes

| Mode | Phase approval | Iteration approval | File writes |
|---|---|---|---|
| `Standard` | Prompts user (auto-approves if <= threshold files changed) | Always continues | Allowed |
| `Autonomous` | Auto-approves | Auto-continues; stale detection active | Allowed |
| `Readonly` | Auto-approves | Always continues | Blocked; error if writes occur |

### Key Types

**`GateDecision`** — `Approved`, `ApprovedAll` (equivalent to `--yes`), `Rejected`, `Aborted`.

**`IterationDecision`** — `Continue`, `Skip`, `StopPhase`, `Abort`.

**`SubPhaseSpawnDecision`** — `Approved`, `Skipped`, `RejectAll`.

**`ApprovalGate`** — interactive gate for `Standard` mode. `check_phase()` renders phase header, shows previous changes, dispatches based on `permission_mode`. Uses `dialoguer::Select` for prompts. `skip_all` is set by the `--yes` flag or user selecting "yes to all."

**`ProgressTracker`** — tracks stale progress. `update(changes, progress_pct)` increments `stale_iterations` when neither new files appear nor the progress percentage increases. `is_making_progress(threshold)` returns false when `stale_iterations >= threshold`.

**`AutonomousGateStrategy`** — non-interactive gate. Always approves phases and iterations. On stale via `check_stale_progress()`: first stale sets `pivot_issued = true`, resets counter, returns `Continue` (caller injects `DEFAULT_PIVOT_PROMPT`). Second stale returns `StopPhase`.

The `DEFAULT_PIVOT_PROMPT` is intentionally harsh — "CRITICAL: You have made no progress… try a fundamentally different strategy" — to force Claude to abandon a failing approach.

### Readonly Mode Enforcement

`validate_readonly_changes(phase, changes)` errors with `"Readonly mode violation"` if any files were added or modified. The orchestrator fails the phase immediately on this error.

### Sub-Phase Spawn Gating

`check_sub_phase_spawn()` validates requested budget against remaining budget. In `Standard` mode it prompts; in `Autonomous` mode it auto-approves within budget, auto-skips over budget.

### Connections

- Main phase loop — calls `check_phase()` before each phase, `check_stale_progress()` after stale iterations.
- `src/phase.rs` — `Phase.permission_mode` drives which strategy is used.
- `src/audit/mod.rs` — `FileChangeSummary` feeds both `ProgressTracker::update()` and `validate_readonly_changes()`.

## Section 10: Git Tracking & Audit

**Files:** `src/tracker/mod.rs`, `src/tracker/git.rs`, `src/audit/mod.rs`

### Purpose

Two concerns separated here:

- **Git Tracker** (`src/tracker/`) — answers "what changed during this iteration?" by bookending each Claude invocation with a snapshot commit.
- **Audit Trail** (`src/audit/`) — defines the data model for the complete audit log.

### Git Tracker (`src/tracker/git.rs`)

`GitTracker` wraps `git2::Repository` with three operations:

**`snapshot_before(phase) -> Result<String>`** — stages all files and creates a `[forge] snapshot before phase {N}` commit. Handles the unborn-branch case (new repo, no commits) by creating an initial parentless commit. Returns the commit SHA stored as `IterationAudit.git_snapshot_before`.

**`compute_changes(before_sha) -> Result<FileChangeSummary>`** — diffs `before_sha` tree against current workdir+index. Classifies `Added`, `Modified`, `Deleted` files; counts `+`/`-` lines.

**`get_full_diffs(before_sha) -> Result<Vec<FileDiff>>`** — same diff but also captures full unified diff strings via `git2::Patch`. Used by the audit logger.

**`head_sha() -> Option<String>`** — current HEAD SHA; `None` on unborn branches.

The snapshot approach means Forge's git history records the state before every Claude invocation. Operators can `git diff <before_sha> HEAD` to see exactly what a single iteration produced.

### Audit Data Model (`src/audit/mod.rs`)

Pure data definitions. Hierarchy:

```
AuditRun
  └─ Vec<PhaseAudit>
       ├─ Vec<IterationAudit>
       ├─ Vec<CompactionEvent>
       └─ Vec<SubPhaseAudit>
            └─ Vec<IterationAudit>
```

**`AuditRun`** — `run_id: Uuid`, timestamps, `RunConfig`, and the phase list.

**`PhaseAudit`** — per-phase record. `total_iterations()` includes sub-phase iterations. `total_file_changes()` aggregates sub-phases. `all_sub_phases_completed()` checks nested outcomes.

**`PhaseOutcome`** — `InProgress`, `Completed { iteration }`, `MaxIterationsReached`, `Error { message }`, `UserAborted`, `Skipped`.

**`IterationAudit`** — per-invocation record with: `duration_secs`, `claude_session` (file paths, char counts, exit code, token usage, `session_id`), `git_snapshot_before`, `git_snapshot_after`, `file_diffs`, `promise_found`, `signals: Option<IterationSignals>`, `council_data: Option<CouncilAuditData>`. The `session_id` field uses `skip_serializing_if = "Option::is_none"` for backward compatibility with old audit logs.

**`FileChangeSummary`** — `files_added`, `files_modified`, `files_deleted` (`Vec<PathBuf>`) plus line counts. `is_empty()` gates the `IterationFeedback` builder.

**`CompactionEvent`** — records context compaction: iterations compacted, original/summary char counts, compression ratio.

### Connections

- `src/orchestrator/runner.rs` — calls `snapshot_before()` before Claude, `compute_changes()` and `get_full_diffs()` after, then builds `IterationAudit` for `AuditLogger`.
- `src/gates/mod.rs` — `FileChangeSummary` feeds `ProgressTracker::update()` and `validate_readonly_changes()`.
- `src/audit/logger.rs` — writes `AuditRun` to disk as JSON (append-only, not part of checkpoint recovery).
- `src/factory/db/` — stores a lighter version of this data for the Factory UI.
