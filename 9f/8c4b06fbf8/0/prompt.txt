## Review Workflow:

1. **Determine Review Scope**
   - Check git status to identify changed files
   - Parse arguments to see if user requested specific review aspects
   - Default: Run all applicable reviews

2. **Available Review Aspects:**

   - **comments** - Analyze code comment accuracy and maintainability
   - **tests** - Review test coverage quality and completeness
   - **errors** - Check error handling for silent failures
   - **types** - Analyze type design and invariants (if new types added)
   - **code** - General code review for project guidelines
   - **simplify** - Simplify code for clarity and maintainability
   - **all** - Run all applicable reviews (default)

3. **Identify Changed Files**
   - Run `git diff --name-only` to see modified files
   - Check if PR already exists: `gh pr view`
   - Identify file types and what reviews apply

4. **Determine Applicable Reviews**

   Based on changes:
   - **Always applicable**: code-reviewer (general quality)
   - **If test files changed**: pr-test-analyzer
   - **If comments/docs added**: comment-analyzer
   - **If error handling changed**: silent-failure-hunter
   - **If types added/modified**: type-design-analyzer
   - **After passing review**: code-simplifier (polish and refine)

5. **Launch Review Agents**

   **Sequential approach** (one at a time):
   - Easier to understand and act on
   - Each report is complete before next
   - Good for interactive review

   **Parallel approach** (user can request):
   - Launch all agents simultaneously
   - Faster for comprehensive review
   - Results come back together

6. **Aggregate Results**

   After agents complete, summarize:
   - **Critical Issues** (must fix before merge)
   - **Important Issues** (should fix)
   - **Suggestions** (nice to have)
   - **Positive Observations** (what's good)

7. **Provide Action Plan**

   Organize findings:
   ```markdown
   # PR Review Summary

   - [agent-name]: Issue description [file:line]

   ## Important Issues (X found)
   - [agent-name]: Issue description [file:line]
   ```

## Summary

This PR migrates the database layer from synchronous `rusqlite` (behind `Arc<Mutex>` + `spawn_blocking`) to async `libsql` with Turso embedded replica support. The database module is cleanly decomposed from one monolithic `db.rs` into `db/mod.rs`, `db/agents.rs`, `db/issues.rs`, `db/migrations.rs`, `db/pipeline.rs`, `db/projects.rs`, and `db/settings.rs`. ## CRITICAL Issues

### Issue 1: `DbHandle::sync()` silently discards sync errors

**Location:** `/Users/jdsingh/Projects/AI/forge/src/factory/db/mod.rs`, line 86

**Severity:** CRITICAL

**Issue Description:** The `sync()` method uses `let _ = self.db.sync().await;` which explicitly discards the `Result` from the libsql sync operation. The method then unconditionally returns `Ok(())`, making it impossible for callers to know that synchronization with the Turso remote failed. This is the Turso embedded replica's mechanism for pushing local writes to the remote and pulling remote changes. If this fails silently, the local replica could diverge from the remote database with no indication whatsoever.

**Hidden Errors:** Network failures to Turso cloud, authentication token expiration, Turso service outages, TLS errors, connection timeouts, server-side conflicts. Every single one of these will be swallowed.

**User Impact:** Users running Turso embedded replicas in production could experience stale data, lost writes (writes that succeed locally but never propagate to remote), and split-brain scenarios between multiple instances -- all with zero logging or error feedback. Debugging this would be a nightmare because the method signature promises `Result<()>` but never returns `Err`.

**Recommendation:** At minimum, log sync failures. Ideally, propagate the error so callers can decide how to handle it. The caller in `server.rs` line 113 does `db_handle.sync().await?` -- but it will never see an error because the method never returns one.

**Corrected code:**
```rust
pub async fn sync(&self) -> Result<()> {
    self.db
        .sync()
        .await
        .context("Failed to sync embedded replica with Turso remote")?;
    Ok(())
}
```

### Issue 2: `health_check` silently swallows integrity check row-parsing failures

**Location:** `/Users/jdsingh/Projects/AI/forge/src/factory/db/mod.rs`, lines 443-458

**Severity:** CRITICAL

**Issue Description:** The `health_check` function has a triple-nested `if let Ok(...)` pattern that silently discards failures at two levels:
1. `if let Ok(Some(row)) = rows.next().await` -- if `next()` returns `Err(...)`, it's silently dropped
2. `if let Ok(result) = row.get::<String>(0)` -- if the column extraction fails, it's silently dropped

If the integrity check query succeeds but the row parsing fails (e.g., the database returns a non-string value, or the row is malformed), the function silently returns without any warning. The user sees no output and assumes the database is healthy.

**Hidden Errors:** Row deserialization failures, type mismatches, database corruption that causes the integrity check to return malformed data. These are precisely the kinds of errors the integrity check is supposed to surface.

**User Impact:** A corrupted database could pass the health check with zero warnings, leading to data loss or silent corruption propagation.

**Recommendation:** Log every failure branch explicitly.

**Corrected code:**
```rust
pub async fn health_check(conn: &Connection) {
    match conn.query("PRAGMA integrity_check", ()).await {
        Ok(mut rows) => {
            match rows.next().await {
                Ok(Some(row)) => {
                    match row.get::<String>(0) {
                        Ok(result) if result == "ok" => { /* healthy */ }
                        Ok(result) => {
                            eprintln!("[db] WARNING: database integrity check failed: {result}");
                        }
                        Err(e) => {
                            eprintln!("[db] WARNING: could not read integrity check result: {e}");
                        }
                    }
                }
                Ok(None) => {
                    eprintln!("[db] WARNING: integrity check returned no rows");
                }
                Err(e) => {
                    eprintln!("[db] WARNING: failed to read integrity check row: {e}");
                }
            }
        }
        Err(e) => {
            eprintln!("[db] WARNING: failed to run integrity check: {e}");
        }
    }
}
```

## Type Design Analysis: `feat/libsql-turso-migration` Branch

This analysis covers the rewritten database layer that migrates from synchronous `rusqlite` (behind `Arc<Mutex>` + `spawn_blocking`) to native async `libsql` with Turso embedded replica support.

---

## Type: `DbHandle` (src/factory/db/mod.rs)

### Invariants Identified

1. **Database is always initialized**: Pragmas (WAL mode, busy timeout, foreign keys) and migrations are run before any `DbHandle` is returned.
2. **Single shared connection**: All operations share one `Arc<Connection>` to avoid the libsql `:memory:` quirk where each `connect()` call creates a separate database.
3. **Construction-only initialization**: The `from_db` private method is the sole path from a raw `Database` to a `DbHandle`, ensuring init steps cannot be skipped.
4. **Turso vs. local is a construction-time decision**: Once constructed, the mode is fixed. Callers interact uniformly regardless of backend.
5. **`db` field is `Arc<Database>`**: The underlying database handle is reference-counted for cheap cloning.
6. **Sync is a no-op for local**: `sync()` silently swallows the result for local-only databases.

### Ratings

- **Encapsulation**: 6/10
  The `DbHandle` struct fields (`db`, `conn`) are private, which is good. However, the `conn()` method returns `&Connection` directly, fully exposing the raw libsql connection to all callers. Any code with a reference to `DbHandle` can execute arbitrary SQL, bypass all domain-module boundaries, and call `last_insert_rowid()` at any time. The submodule functions (e.g., `agents::create_agent_task`) also accept a raw `&Connection`, so nothing prevents callers from constructing queries outside the defined domain modules. The facade pattern on `DbHandle` (all those convenience methods) is good intent, but undermined by the public `conn()` escape hatch.

- **Invariant Expression**: 7/10
  The type's structure clearly communicates "initialized database handle." The three named constructors (`new_local`, `new_remote_replica`, `new_in_memory`) make the valid construction modes explicit and self-documenting. The doc comments on the struct explain the two modes well. However, the distinction between local-only and remote-replica is invisible after construction -- there is no type-level or runtime way to query which mode you are in, which could matter for diagnostics.

- **Invariant Usefulness**: 8/10
  The "always initialized" invariant is the most valuable property of this type. It eliminates an entire class of "forgot to run migrations" bugs. The shared-connection invariant solves a real, documented libsql footgun with in-memory databases. These invariants are well-targeted at actual production bugs.

- **Invariant Enforcement**: 6/10
  Construction is well-guarded: `from_db` is private, and the three public constructors all go through it. However, the `conn()` method creates a wide-open enforcement gap. Any caller can obtain the connection and call `last_insert_rowid()`, `execute()`, or `transaction()` directly, potentially violating domain invariants. The convenience methods on `DbHandle` are pure pass-throughs that add no validation -- they simply call `self.conn()` and delegate. This means the facade provides no additional safety over raw connection access.

### Strengths

- **Clean migration from sync to async**: The old design used `Arc<Mutex<FactoryDb>>` with `spawn_blocking`, which blocked tokio threads during the lock wait. The new design uses natively async libsql, eliminating both the mutex and the blocking thread hops.

- **Elimination of `lock_sync`**: The old design had a `lock_sync()` method that allowed callers to synchronously block on the mutex -- a dangerous footgun in async code. This is completely gone.

- **Well-structured module decomposition**: The monolithic 2,400-line `db.rs` is broken into focused domain modules (`agents.rs`, `issues.rs`, `pipeline.rs`, `projects.rs`, `settings.rs`, `migrations.rs`), each with their own tests. This is a significant maintainability improvement.

- **Proper migration system**: The old code used `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE` with error swallowing (ignoring "duplicate column" errors). The new code uses a proper versioned migration system with `PRAGMA user_version`, including bootstrap detection for pre-migration databases. This is a substantial improvement in correctness.

- **Good error context**: Every database operation uses `.context()` to provide human-readable error messages, making failures debuggable.

- **Comprehensive test coverage**: Every submodule has targeted async tests using `new_in_memory()`.

### Concerns

1. **`last_insert_rowid()` race condition with shared connection**: This is the most significant correctness concern. The `conn` is `Arc<Connection>` shared across all async tasks. Functions like `create_project`, `create_issue`, `create_agent_team`, `create_agent_task`, and `create_agent_event` all follow this pattern:

   ```rust
   conn.execute("INSERT ...", params).await?;
   let id = conn.last_insert_rowid();
   ```

   With a shared connection and concurrent requests (e.g., two API requests creating issues simultaneously), there is a window between `execute` and `last_insert_rowid` where another insert on the same connection can change the rowid. libsql's `last_insert_rowid()` is per-connection, not per-statement. Since all operations share one connection, concurrent inserts will race.

2. **`create_issue_from_github` has a TOCTOU race**: The function checks `SELECT COUNT(*) > 0 ... WHERE github_issue_number = ?` and then does an `INSERT` in two separate operations without a transaction. Under concurrent access, two requests with the same `github_issue_number` could both pass the check and both insert, violating the unique index (which would produce a database error rather than a graceful `None` return).

3. **`create_agent_team` is not transactional**: It performs an `INSERT INTO agent_teams` followed by `UPDATE pipeline_runs SET team_id = ?1`, but these are two separate statements with no transaction. If the server crashes between them, the team exists but the pipeline run does not reference it. The function also constructs the return value with `chrono::Utc::now().to_rfc3339()` rather than reading back the database's `datetime('now')`, creating a potential timestamp mismatch.

4. **Dual `upsert_pipeline_phase` methods**: Both `upsert_pipeline_phase` (takes `&str` for status) and `upsert_pipeline_phase_typed` (takes `&PhaseStatus`) exist. The untyped version accepts arbitrary strings for the status column, bypassing the `PhaseStatus` enum entirely. All external callers use the untyped version (line 1896, 1931, 1966 in pipeline.rs), so the type safety provided by `PhaseStatus` is not actually utilized in practice.

5. **`update_issue` validates priority but not labels**: The `update_issue` function validates the `priority` parameter by parsing it through `Priority::from_str`, but the `labels` parameter is passed directly to the database as a raw string with no validation that it is valid JSON. If a caller passes `"not json"`, it will be stored and cause deserialization failures on read.

6. **`sync()` swallows errors silently**: The `sync` method does `let _ = self.db.sync().await;` and always returns `Ok(())`. If Turso sync fails, the caller has no way to know. This is documented as "no-op for local-only" but the error swallowing applies to remote replicas too, where a sync failure is operationally significant.

7. **Duplicated `row_to_pipeline_run` function**: The `row_to_pipeline_run` helper is defined identically in both `issues.rs` (line 37) and `pipeline.rs` (line 8). The `RUN_COLS` constant is also duplicated. This is a maintenance hazard -- if the schema changes, both copies must be updated.

8. **Column index fragility**: All `row_to_*` functions use positional indexing (e.g., `row.get(0)?`, `row.get(4)?`). If a `SELECT` column list changes, every index must be manually re-counted. This is a common source of subtle bugs after schema changes.

### Recommended Improvements

1. **Address the `last_insert_rowid` race**: Wrap insert-then-fetch patterns in transactions, or use `RETURNING id` if libsql supports it. Alternatively, consider using one connection per logical operation via `db.connect()` for file-backed databases (only the in-memory case requires a shared connection). A pragmatic middle ground: use transactions for all insert operations.

2. **Remove the public `conn()` method**: Replace it with purpose-specific methods or, at minimum, rename it to `conn_raw()` and document that it should only be used from the `db::*` submodules. Making it `pub(crate)` or `pub(super)` would be even better, restricting access to just the db module tree.

3. **Eliminate the untyped `upsert_pipeline_phase`**: The callers in `pipeline.rs` already have access to status strings that should be parsed into `PhaseStatus`. Force all callers through the typed variant. If raw strings from external sources must be supported, parse them at the boundary and fail early.

4. **Deduplicate `row_to_pipeline_run` and `RUN_COLS`**: Move these to a shared location (e.g., a `db/common.rs` or the `mod.rs` file itself) so there is a single source of truth.

5. **Wrap `create_issue_from_github` in a transaction**: The check-then-insert pattern needs atomicity. Alternatively, rely on the unique index by doing an `INSERT ... ON CONFLICT DO NOTHING` and checking the rows affected.

6. **Validate labels JSON in `update_issue`**: Before writing to the database, parse the labels string as `serde_json::Value` (or `Vec<String>`) and reject invalid input. This matches the existing pattern of validating priority.

7. **Make `sync()` return meaningful errors for remote replicas**: At minimum, log the error. Ideally, propagate it so callers can react to sync failures.

## Type: Event Batching Mechanism (src/factory/agent_executor.rs, lines 333-391)

### Invariants Identified

1. **Events are eventually persisted**: The batching mechanism guarantees all events are flushed either when the batch reaches 25 items, when 2 seconds elapse, or when the channel closes.
2. **Events are written in order**: The batch drains in FIFO order.
3. **Channel closure triggers final flush**: When the sender is dropped, all remaining events are flushed before the writer task exits.

### Ratings

- **Encapsulation**: 3/10
  The event batching logic is implemented as an ad-hoc inline closure inside `run_task`, using raw tuples `(i64, String, String, Option<serde_json::Value>)` as the event representation. There is no dedicated type -- it is not reusable, testable in isolation, or configurable. The batch size (25) and flush interval (2 seconds) are hardcoded magic numbers.

- **Invariant Expression**: 4/10
  The batching invariants are encoded in control flow (the `tokio::select!` loop) rather than in types. A reader must trace through the loop logic to understand the flushing guarantees. The tuple type gives no indication of what each field represents.

- **Invariant Usefulness**: 7/10
  The invariants are practical and solve a real problem: agent processes emit hundreds of events per second, and writing each one individually would saturate the database. The time-based flush ensures events are visible in the UI within 2 seconds.

- **Invariant Enforcement**: 5/10
  The `flush_events` function individually inserts each event in the batch without any batching at the SQL level (no transaction, no multi-row insert). ### Strengths

1. **Not a reusable type**: The batching logic is embedded in a method body. If another part of the system needs similar batching, it would have to be copy-pasted.
2. **No SQL-level batching**: The `flush_events` function loops over the batch and calls `create_agent_event` once per event. A transaction wrapping the batch would provide atomicity and better performance.

### Recommended Improvements

1. **Extract a `EventBatcher` struct**: Encapsulate the channel, flush logic, and configuration (batch size, interval) into a reusable type with clear construction and `async fn run()` method.
2. **Use a named struct for pending events**: Replace the 4-tuple with a struct.
