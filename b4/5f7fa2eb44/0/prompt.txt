The `_phase_span` and `_iter_span` variables use `.entered()` which creates a `tracing::span::EnteredSpan`. This type uses thread-local storage to set the current span. When the future is polled on different threads (which is normal for tokio's multi-threaded runtime), the span context will be wrong or will panic. The tracing docs explicitly warn against this: "An entered span must not be moved to another thread or across await points".

**CLAUDE.md rule:** "Use tokio for all async -- never block in async code" (this is related -- `.entered()` uses thread-local state that is incompatible with async task migration).

**Fix:** Replace `.entered()` with `Instrument` from `tracing::Instrument`:

```rust
// Instead of:
let _phase_span = info_span!("phase", ...).entered();

// Use (wrapping the async block):
async { /* body */ }.instrument(info_span!("phase", ...)).await
```

## Type: `MetricsCollector` (`src/metrics/mod.rs`)

### Invariants Identified
- Holds a valid `DbHandle` to a database where the metrics tables exist
- All metrics recording requires a valid `run_id` to be inserted first (foreign key dependency)
- Phase metrics require a run to exist; iteration/review/compaction metrics require a phase to exist
- The lifecycle of a run follows: `record_run_started` -> (phases) -> `record_run_completed`
- The lifecycle of a phase follows: `record_phase_started` -> (iterations, reviews) -> `record_phase_completed`
- Query methods require `days > 0` for meaningful results

### Ratings
- **Encapsulation**: 6/10
  The `DbHandle` field is private, which is good. However, `MetricsCollector` is essentially a thin wrapper around raw SQL strings, exposing no domain abstraction. The `DbHandle` itself is `Clone + Arc`-wrapped, so the same `FactoryDb` is shared with `AppState.db`, meaning any code with access to the `DbHandle` can bypass the collector entirely and run arbitrary SQL against the metrics tables. Additionally, `FactoryDb.conn` was changed from private to `pub(crate)`, weakening the encapsulation of the database connection itself.

- **Invariant Expression**: 3/10
  This is the weakest aspect. Nearly every parameter that has a finite set of valid values is represented as either `&str` or bare `i32`/`f64`. The `outcome` parameter in `record_phase_completed` accepts an arbitrary `&str`, yet the codebase already has a `PhaseOutcome` enum. The `specialist_type` and `verdict` parameters accept arbitrary strings, yet `ReviewVerdict` and specialist type enums exist. The `success` field is a `bool` when passed to `record_run_completed` but stored as an `i32` in SQLite. The `phase_number` is a plain `i32` but elsewhere in the codebase it is a `String` (the `PhaseNumber` pattern). The method signatures have so many positional parameters of the same type (multiple `i32` values in sequence) that callers can easily swap arguments without any compile-time warning.

- **Invariant Usefulness**: 6/10
  The foreign key constraints in the database schema do enforce some ordering invariants (a phase must belong to a run, an iteration must belong to a phase). The `UNIQUE(run_id, phase_number)` constraint prevents duplicate phases. These are genuine, useful constraints. However, they are runtime-only (SQLite errors), not compile-time.

- **Invariant Enforcement**: 4/10
  There is no validation at construction time beyond receiving a `DbHandle`. There is no builder pattern or typed event structure for recording metrics. The lifecycle ordering (run must be started before phases, phases before iterations) is enforced only by the database foreign keys, meaning violations surface as opaque SQLite errors rather than clear domain errors. The `days` parameter for queries is `u32`, which at least prevents negative values, but `0` is technically valid and semantically meaningless. No upper bound is checked for `limit` or `days`.

### Strengths
- The async design using `DbHandle::call` is consistent with the rest of the codebase
- Test coverage is thorough, including lifecycle tests and empty-state edge cases
- Query methods return well-typed response structs rather than raw tuples
- The use of `Result<T>` with `.context()` follows project conventions

1. **Stringly-typed domain values**: `outcome`, `specialist_type`, and `verdict` are all `&str` parameters where the codebase already has enums (`PhaseOutcome`, `ReviewVerdict`). This means the metrics layer is disconnected from the type system of the domain it measures.

## Cross-Cutting Concerns

### The Stringly-Typed Boundary

The most significant design weakness across the entire change set is the pervasive use of raw strings where domain enums already exist. The codebase has:

- `PhaseOutcome` enum with `Completed`, `MaxIterationsReached`, `Error`, `UserAborted`, `Skipped` variants
- `ReviewVerdict` enum with `Pass`, `Warn`, `Fail` variants
- Specialist types as structured data in the review system

Yet `MetricsCollector` accepts all of these as `&str`. This creates a parallel, untyped data model for the metrics layer. If someone renames a `PhaseOutcome` variant, the metrics SQL strings will silently diverge. If a new `ReviewVerdict` variant is added, the metrics layer has no compiler nudge to handle it.

### Lifecycle Protocol Not Encoded in Types

The temporal ordering requirement (start run -> start phase -> record iterations -> complete phase -> complete run) is a critical invariant that exists only in the caller's discipline and the database foreign keys. A state machine pattern -- where `record_run_started` returns a `RunHandle` that exposes `start_phase`, which returns a `PhaseHandle` that exposes `record_iteration` -- would make it impossible to record an iteration before starting a phase at compile time.

### TypeScript/Rust Type Drift Risk

There is no shared schema or code generation between the Rust response types and the TypeScript interfaces. The types are manually duplicated. Changes to `queries::SummaryStats` in Rust will not cause any TypeScript compile error. For a metrics dashboard, where data integrity is the entire point, this is a notable gap.
